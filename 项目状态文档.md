# DeltaForge v6.1 — 2026年度全模块审计报告

> 审计时间: 2026-07-24 | 审计标准: 2026 Android 反检测主流技术
> 仓库: github.com/wzy887011/DeltaForge | 分支: master
> 基于对 2026 年业内 Zygisk-Loader / fake-linker / xHook / bhook / memfd injection / mremap hiding / COPG 自卸载 等前沿方案的对比审计

---

## 总体评估

| 维度 | 当前状态 | 2026标准 | 差距 |
|------|---------|---------|------|
| 注入方式 | ptrace (forge -l) + so替换 (hijack) | Zygisk+memfd 或 ptrace+memfd | **hijack必须弃用**, ptrace需加固 |
| SO隐藏 | madvise(MADV_DONTDUMP) | mremap匿名重映射 | 不防/proc/self/maps扫描 |
| libc Hook | 16函数覆盖+延迟激活 | PLT/GOT hook + 延迟激活 | 覆盖面好，激活时机可优化 |
| 属性伪装 | hook + JNI Build + setprop | 跨层一致性(kernel→HAL→framework) | JNI RegisterNatives替换整表有风险 |
| 内存补丁 | 硬编码偏移 67+40+6处 | 特征码扫描自动定位 | **游戏更新即失效** |
| 文件清理 | 定时删除 | 伪造文件内容替代删除 | 文件被反复重建 |
| 网络过滤 | DNS+IP黑名单 | 动态更新 | 列表可能过期 |
| GPU伪装 | **已禁用** | 运行时查询后匹配目标 | **致命缺口** |
| BPF | exit_group (调试禁用中) | exit_group + tgkill白名单 | 当前禁用 |
| 部署工具链 | shell脚本，git pull 易冲突 | 版本校验+回滚 | 脆弱 |

## 逐模块详细审计

### 1. 注入机制 (injector.c + forge.c + libforgehook.c)

**2026 主流方案:**

| 方案 | 隐蔽性 | 复杂度 | 本项目 |
|------|--------|--------|--------|
| Zygisk + memfd (Zygisk-Loader v2.0) | ★★★★★ | 中 | 未实现 |
| Ptrace + memfd + 延迟注入 | ★★★★ | 低 | 接近 (forge -l) |
| Ptrace + 磁盘.so | ★★★ | 低 | 当前实现 |
| So文件替换 (hijack) | ★ | 高 | **应弃用** |

**发现问题:**
- `injector.c` 的 ptrace attach 可被 anti-ptrace 检测 (TracerPid)
- 注入后 so 在 /proc/self/maps 中可见文件路径 — 可被扫描发现
- hijack 模式的 so 替换方案存在架构性缺陷 — tersafe 完整校验 + dt_needed 缺失的双重打击
- `forge_monitor.c:168-179` 有 check_tracer() 但只是监控别人，没隐藏自己

**建议:**
- **P0: 弃用 hijack，专注 forge -l inject 模式**
- **P1: injector 改为 memfd_create 加载，零磁盘痕迹**
- **P1: 注入后用 mremap 隐藏 maps 条目**

### 2. SO 隐藏 (libforgehook.c constructor 50 + 101)

**当前:** `_hide_self_from_maps()` → madvise(MADV_DONTDUMP)，仅防 dump，不防扫描
**2026标准:** mremap 匿名重映射 — 把文件映射替换为 `[anon]` 彻底消失

**注意 2026 反制:** mremap 会留下匿名 r-xp 段，反作弊可扫描 maps 中的匿名可执行段检测

**建议:**
- **P1: 实现 mremap 匿名重映射替代 madvise**
- **P1: 伪造 maps fd 中同时过滤匿名 r-xp 痕迹**
- **P2: 配合 soinfo 链表隐藏 (dl_iterate_phdr 过滤已实现)**

### 3. libc Hook 延迟激活 (libforgehook.c)

**当前:** 16函数全覆盖 + g_hooks_ready 延迟激活
**激活时机:** tersafe patch 线程找到 libtersafe.so 后 (constructor 150 启动的线程)
**覆盖面:** open/fopen/openat/access/stat/lstat/readlink/readlinkat/tgkill/kill/exit_group/getaddrinfo/connect/getenv/dlopen/dladdr/dl_iterate_phdr/opendir/readdir/readdir64/__system_property_get

**缺陷:** 激活前 hook 纯透传 — 游戏启动阶段的所有文件/属性访问无保护

**建议:**
- **P1: 激活前将文件访问记录到延迟审计日志 (buffer + flush on activation)**
- **P2: 审计 HIDDEN[] 和 OVERRIDE_FILES[] 是否覆盖所有检测文件**

### 4. 属性伪装 (libforgehook.c + forge.c)

**当前三层:** __system_property_get hook + JNI Build覆盖 + setprop/resetprop
**2026 检测:** 跨层交叉验证 — 游戏同时查 Java / native / /system/build.prop → 不一致则检测到

**关键缺陷:** `jni_hook_system_properties` 用 `RegisterNatives` 替换了 SystemProperties 的**整个** native 方法表 — 如果原版 qimei 也注册了方法就冲突

**建议:**
- **P1: JNI SystemProperties hook 改为选择性的，不替换全表**
- **P2: 扩展 BUILD_FIELDS 覆盖全部可能被检查的字段**

### 5. 内存补丁 (forge.c)

**当前:** 67处 tersafe代码 + 40处 BSS + 6处 UE4 + kill链6节点，全部硬编码偏移

**致命缺陷:** 硬编码偏移 — libtersafe.so 或 libUE4.so 更新后全部失效

**2026 标准:** 特征码扫描 (pattern scan) — 定位后计算偏移 → 跨版本可用

**建议:**
- **P1: kill chain 6 节点改用特征码扫描 (最关键的6处)**
- **P2: BSS 偏移改为 ELF 符号表解析**
- **P3: 补丁健康定期报告**

### 6. 文件清理 (forge.c)

**当前:** 删 qm/ano_tmp/tdm_tmp/crashsight 等目录 + 定时守护
**缺陷:** 文件被删后游戏立即重建 — CPU 空转

**建议:**
- **P2: 不删文件，改为伪造文件内容为空白合法格式**
- **P2: 注入前清除文件，减小窗口期**

### 7. 网络过滤 (libforgehook.c)

**当前:** getaddrinfo hook (14域名) + connect hook (5 IP段)
**缺陷:** 只过滤 TCP ESTABLISHED，UDP 未覆盖

**建议:**
- **P2: 添加 UDP sendto hook**
- **P2: 域名/IP 列表动态更新**

### 8. GPU 伪装 (libforgehook.c constructor 120)

**当前状态: 已禁用** (返回固定 "Adreno (TM) 740" 字符串，云手机虚拟GPU是Mali，功能不匹配导致黑屏)

**2026 正确做法:**
1. 查询云手机真实 GPU: `glGetString(GL_RENDERER)` → 得到真实硬件能力
2. 选择与真实 GPU 能力匹配的目标设备
3. 返回一致且合理的目标设备字符串和扩展列表
4. 确保 GL_EXTENSIONS 返回的扩展真实存在

**建议:**
- **P1: 运行时查询真实GPU → 匹配目标 → 动态生成匹配的假字串**

### 9. BPF Seccomp (libforgehook.c constructor 49)

**当前: 调试禁用中** — 临时注释以排查闪退原因
**v6.1 版本:** exit_group-only (7条指令)，已被验证运行

**建议:**
- **P0: 恢复 exit_group-only BPF**
- **P2: 添加 tgkill/kill 白名单 (SIGUSR1/SIGUSR2/SIGQUIT)**

### 10. 进程监控 (forge_monitor.c)

**评估:** 功能完整 — inotify + fd扫描 + 网络扫描 + TracerPid检查 + audit尾读。作为诊断工具合理，不需大改。

### 11. 触控注入 (touch_injector.c)

**评估:** /dev/uinput I2C 触摸模拟，伪装成 "fts_ts" 触摸屏，绕过 InputDispatcher 检测。合理，不需大改。

### 12. 脚本工具链 (deploy.sh / check.sh / df-hijack-root.sh 等)

**缺陷:**
- deploy.sh 的 hijack 更新在 cp 操作中可能因 "Text file busy" 失败
- git pull 与本地 sed 修改冲突频繁
- check.sh 日志路径有时读错 (app data vs /data/local/tmp)

**建议:**
- **P2: deploy.sh 加 --skip-hijack 选项**
- **P2: 加 md5 预检，一致时跳过 hijack 更新**
- **P2: check.sh 输出 JSON 格式**

## 核心结论

**DeltaForge v6.1 的架构方向是对的** — ptrace注入+内存补丁+libc hook+属性伪装+文件清理+网络过滤。这就是2026年游戏过检测的标准方案。

**hijack (so替换) 必须彻底弃用。** 它在架构上与 tersafe 的完整性校验无法共存，2026年也没有任何人用这种方案做游戏mod。

**GPU伪装是最紧迫的功能需求。** 当前黑屏说明UE4渲染器初始化失败，这意味着云手机的真实GPU能力与伪造的不匹配。必须查询真实GPU后选择匹配的目标设备。

**内存补丁的硬编码偏移是最大维护风险。** 游戏更新后67处补丁全部失效，必须引入特征码扫描机制。
